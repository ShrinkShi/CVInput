# 关于子窗口出现位置反复修不好的经验总结

## 背景

CVInput 的设置窗口、关于窗口需要贴在主窗口左侧显示：

```text
子窗口右边缘 + gap = 主窗口左边缘
```

理论公式很简单：

```python
child_x = main_x - child_width - gap
```

但在 Windows 150% 缩放下，子窗口一直和主窗口重叠。100% 缩放基本正常，所以问题一开始看起来像 DPI 坐标系混乱。

最终证明：真正的问题不是左侧贴靠公式本身，而是 **customtkinter / Tk 在非 100% 缩放下，geometry 里的逻辑尺寸和实际视觉尺寸不是同一个东西**。

## 之前修过什么

### 1. 修左侧贴靠公式

最开始怀疑公式写错，确认正确公式应该是：

```python
child_x = main_x - child_width - gap
```

并且左侧空间不足时放到右侧：

```python
if child_x < 0:
    child_x = main_x + main_w + gap
```

这个方向是必要的，但不是最终根因。

### 2. 统一 Tk 坐标系

后来怀疑混用了：

- `winfo_x()`
- `winfo_rootx()`
- `geometry("+x+y")`
- Windows 物理像素
- Tk 逻辑像素

于是把定位函数尽量收敛到 Tk 逻辑坐标，避免 Win32 `GetWindowRect` / `SetWindowPos` 这类物理像素接口参与定位。

这个方向也合理，但仍没有解决，因为当时忽略了一个更隐蔽的问题：**坐标 x/y 是一致的，但宽高不是视觉宽高**。

### 3. 修 Toplevel 首次 133x133

日志显示首次设置：

```text
target_geometry='344x486+626+120'
after child_geometry='133x133+626+120'
```

也就是说，`CTkToplevel` 在 `withdrawn` 状态下不会立刻结算真实尺寸，会临时显示成 `133x133`。

于是加过：

- 创建时 `withdraw()`
- `alpha=0`
- 延迟 `after`
- 等尺寸稳定后再显示
- 透明预映射

这解决了“首次尺寸未稳定”的一部分问题，但依然没有解决最终重叠。原因是：即使窗口最后变成了 `344x486`，这个 `344` 仍然只是 geometry 逻辑宽度，不是屏幕上的视觉宽度。

### 4. 启动时声明 DPI awareness

又怀疑进程没有声明 DPI-aware，导致 Windows 做 DPI 虚拟化。

于是加了：

```python
SetProcessDpiAwarenessContext(-4)  # PER_MONITOR_AWARE_V2
SetProcessDpiAwareness(2)
SetProcessDPIAware()
```

这个是正确的基础设施补强，但也没有直接解决窗口重叠，因为 customtkinter 自己还有缩放层。

### 5. 增加正式调试日志

最后增加了可开关的调试日志系统，记录：

- 主窗口 `geometry`
- 主窗口 `winfo_width`
- 子窗口目标 geometry
- 子窗口实际 geometry
- `winfo_x/rootx`
- 计算出的 `child_x`

这一步是关键转折点。没有日志时一直在猜，有日志后终于能看到矛盾点。

## 为什么之前修不好

之前一直盯着“坐标是否正确”，但真正出问题的是“宽度取错了”。

日志里最关键的一行是：

```text
geometry='380x210+980+120'
winfo_width=570
```

这说明在 150% 缩放下：

```text
geometry 宽度 = 380
实际视觉宽度 = 570
570 / 380 = 1.5
```

也就是说：

- `geometry()` 里的 `380` 是逻辑尺寸
- `winfo_width()` 里的 `570` 是实际视觉尺寸

之前公式用的是：

```python
child_x = main_x - 344 - 10
```

但 150% 缩放下，子窗口视觉宽度其实是：

```python
344 * 1.5 = 516
```

所以原计算会少减：

```text
516 - 344 = 172px
```

这正好表现为子窗口向右侵入主窗口，形成重叠。

之前修不好的核心原因：

1. 以为是 x/y 坐标混用，其实 x/y 基本是对的。
2. 以为是首次 Toplevel 尺寸未稳定，其实那只是附带问题。
3. 以为 DPI awareness 能统一所有东西，但 customtkinter 还有自己的缩放表现。
4. 没有同时记录 `geometry()` 和 `winfo_width()`，导致看不到逻辑尺寸与视觉尺寸的差异。

## 这轮为什么修好了

最终修复点在 `src/ui/window_utils.py`。

原来计算左侧贴靠时使用逻辑宽度：

```python
child_x = main_x - child_width - gap
```

现在先计算当前窗口的视觉缩放比例：

```python
scale_x = parent.winfo_width() / geometry_width
```

再把子窗口逻辑宽度换算成视觉宽度：

```python
child_visual_w = child_width * scale_x
```

最后用视觉宽度计算位置：

```python
child_x = main_x - child_visual_w - gap
```

以日志中的 150% 缩放为例：

```text
main_x = 980
child_width = 344
scale_x = 1.5
child_visual_w = 516
gap = 10

child_x = 980 - 516 - 10 = 454
```

这个位置在视觉上才是真正贴到主窗口左边。

右侧 fallback 也需要同理，不能用逻辑宽度：

```python
child_x = main_x + main_visual_w + gap
```

否则主窗口贴近屏幕左边时，子窗口虽然会跑到右边，但仍会留下异常空隙。

## 最终经验

### 1. Windows DPI 问题不要只看坐标，还要看尺寸

定位问题通常会先怀疑 `x/y`，但在 Tk/customtkinter 里，非 100% 缩放下更容易出问题的是：

```text
geometry 逻辑尺寸 != winfo 实际尺寸
```

以后遇到类似问题，必须同时打印：

```python
window.geometry()
window.winfo_x()
window.winfo_y()
window.winfo_rootx()
window.winfo_rooty()
window.winfo_width()
window.winfo_height()
```

只看 `geometry()` 会误判。

### 2. `geometry()` 适合设置窗口，不一定适合判断视觉占位

`geometry("344x486+626+120")` 确实设置成功了。

但屏幕上它可能占据：

```text
516x729
```

所以布局贴靠时不能只用 `344`。

### 3. withdrawn 状态下的 Toplevel 尺寸不可信

`CTkToplevel` 在 `withdrawn` 状态下可能长期报告：

```text
133x133
```

不要指望在 withdrawn 状态下等它自然变成目标尺寸。需要 map 后再检查，或者透明 map 再定位。

### 4. 不要靠固定偏移修 DPI 问题

这次如果硬加 `-172px`，在 100% 缩放会错，在 125% 缩放也会错。

正确做法是从运行时数据计算：

```python
scale_x = winfo_width / geometry_width
```

然后按比例换算。

### 5. 先做日志系统，再继续修复杂 UI 问题

这次真正推进问题的是日志，不是继续猜。

尤其是这些字段直接暴露了根因：

```text
geometry='380x210+980+120'
winfo_width=570
```

以后遇到 UI 坐标问题，先把相关坐标源一次性打全，再判断。

## 当前相关文件

- `src/ui/window_utils.py`
  - 子窗口位置计算
  - 当前缩放比例推导
  - 左侧 / 右侧贴靠计算

- `src/ui/window_mixin.py`
  - 子窗口创建、显示、跟随移动
  - 调用 `left_attached_geometry`

- `src/debug_logger.py`
  - 调试日志开关
  - 内存日志
  - 导出日志

## 一句话总结

这次不是“窗口坐标算错了”，而是“拿逻辑宽度去贴实际视觉宽度”。最终修好的关键，是把子窗口定位计算从逻辑宽度改成了按 customtkinter/Tk 当前缩放换算后的视觉宽度。
